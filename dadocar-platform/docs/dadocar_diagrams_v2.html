O que mudou desde a versão anterior

O webclient virou um produto fechado, autenticado e multi-tenant ("Placas360"). Destaques desta versão:

  • Webclient fechado e multi-tenant — "Placas360": autenticação própria (senha via scrypt, sessão em cookie httpOnly assinado por HMAC), cadastro restrito por código de convite (1º usuário = admin; admins geram convites em /admin/convites), e isolamento por tenant via owner_id em todas as tabelas — cada cliente só vê os próprios dados. Papel Master (admin@3ahub.com.br) enxerga todas as consultas. middleware.ts protege todas as rotas. ADR 0007.
  • "Chassi com Recall?" com IA — em cada consulta CheckTudo o chassi é comparado às faixas de recall pela Anthropic (claude-sonnet-4-6); veredito SIM/NÃO + motivo são persistidos e exibidos no relatório (hover mostra a justificativa). Requer ANTHROPIC_API_KEY.
  • Gráfico de variação FIPE (12 meses) ao passar o mouse em "FIPE / Preço", renderizado direto do JSON (historicoPreco) — sem custo de API.
  • Cache agora é indefinido — consultas KBB e CheckTudo ficam guardadas até limpeza manual (sem expiração de 90 dias).
  • Navegação enxuta + rebrand — abas reduzidas a Tabela KBB (com histórico embutido) e Checa Tudo (+ Convites p/ admin); "Carros Ativos", "Buscar", "Relatórios" e "+ Adicionar Veículo" ocultos. Marca Placas360, tema corporativo; placa renderizada como crachá real (Mercosul vs. modelo antigo) conforme o formato.
  • Integração CheckTudo — terceira Function App independente (dadocar-dev-func-checktudo-brs, Linux Y1 Node 22, plano de consumo próprio no mesmo resource group): dados veiculares por placa via API assíncrona da CheckTudo (login → POST /api/query/order → poll GET /api/query/json-response/:id). Produto selecionável por querycode (66 Veículo Total padrão, 67 Essencial, 13 Decod+Precificador, 71 Dados Cadastrais, 76 Decod+FIPE, 241 Decodificador V.4). Credenciais em KV (checktudo-username, checktudo-password). Nova página /checktudo no menu (ao lado de "Histórico KBB") com seletor de produto, renderização por seções e cache 90 dias em checktudo_consultas (chave placa+produto). ADR 0006.
  • Histórico de consultas KBB + cache 90 dias em Azure SQL — nova tabela kbb_consultas em carros_ativos_db guarda cada consulta Molicar (payload completo + placa, marca, modelo, ano, preço justo, preço Molicar). O server action de /precos serve do cache antes de chamar a função; nova página /historico-kbb no menu lista as consultas como cards expansíveis com deep link de volta. ADR 0005.
  • Segunda Function App — pricing aggregator KBB/Molicar (dadocar-dev-func-pricing-brs, Linux Y1 Node 22): rotas /api/pricing/plate/{plate} e /api/pricing/vin/{vin}, OAuth2 client_credentials contra Molicar, secrets em KV (molicar-client-id, molicar-client-secret); página /precos no webclient renderiza preços por canal (NewVehicle, UsedDealer, SellPrivateParty, SellDealer, FPP). ADR 0004.
  • APIM agora é a porta de entrada do cliente — produto dadocar-beta, API dadocar-vehicle-api em /v1, primeira subscription moneycar-tenanta-beta emitida
  • Gateway logs APIM → Log Analytics (ApiManagementGatewayLogs) — base para billing por subscription
  • Customer-CRM simulador (webclient-fipe.vercel.app) já consome via APIM (Ocp-Apim-Subscription-Key)

Ready to Market? — o que falta para vender, escalar, estabilizar, proteger e lucrar

O produto funciona ponta a ponta e o acesso já é fechado e isolado por cliente. Ainda não é vendável sozinho: faltam pagamento, autocadastro, CRM e um ambiente de produção endurecido/observável. Caminho crítico para a primeira receita: 002 → 003 → 016 → 017.

  • 💰 Vendável — paywall + autocadastro + provisionamento automático de tenant + modelo de cliente + API medida (APIM). Itens 002, 003, 016, 017, 001. (Controle de acesso e isolamento por tenant já feitos — ADR 0007.)
  • 📈 Escalável — RU/autoscale Cosmos (007), cache de token compartilhado (010), pipeline de dados (008).
  • 🟢 Estável — CI/CD (004), alertas + SLOs (005), ambiente de produção (014), DR multi-região (015), OTel (013).
  • 🔒 Seguro — hardening de rede / private endpoints (006), de-identificação LGPD (009). (Auth + segredos já tratados — ADR 0007.)
  • 🤑 Lucrativo — CRM com pagamentos, custo por consulta (Molicar / CheckTudo / Anthropic) e margem por cliente (016).

2. Request Flow Canônico (plate lookup com aggregator)

Fluxo atual end-to-end: Browser → Vercel proxy → APIM gateway (/v1) → Function App → providers em paralelo → resposta unificada { query, generated_at, ran_providers, sources: [...] }. APIM autentica via Ocp-Apim-Subscription-Key, injeta x-functions-key no hop interno; o cliente nunca vê a function-key. Cache-aside no Cosmos faz HIT em ~500ms warm; MISS faz fan-out e popula cache fire-and-forget.

3. Cache-aside Flow — Os 3 Estados de uma Plate Lookup

Implementação atual via src/lib/cache.js integrada em vehicleLookup.js. Cache PK /placa com TTL 30d. Wall time: HIT ~500ms · MISS warm ~700ms · MISS cold start 3-5s. Modo ?fresh=1 ainda planejado.

HIT — wall time ~500ms · zero Infocar call
MISS — fan-out + cache populate fire-and-forget
fresh=1 — planejado · skip cache read

4. Data Flow Atual — Onde Cada Dado Vive

Status real de persistência hoje. Diferente da versão anterior, vários fluxos antes "futuros" já estão wired (cache plate, secrets KV, query events). Apenas VIN cache e EH Capture continuam como planejados.

5. Workflow — Como Adicionar um Novo Provider

Padrão de extensão #1 (mais comum). O provider contract está documentado em _types.js. Três coisas mudam: Key Vault, novo módulo provider, registro no index. Nada nas HTTP triggers muda.
